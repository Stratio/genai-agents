# Skill: DOCX Reader

A disciplined approach to getting useful content out of Word documents. Different files need different tactics — a clean prose report behaves nothing like a heavily-reviewed document full of tracked changes, and neither behaves like a legacy `.doc` exported from Office 2003. Diagnose first, extract second.

## 1. Two modes: quick and deep

This skill supports two ways of working:

**Quick mode — `scripts/quick_extract.py`**
One-shot extraction that returns structured Markdown. Use it when:
- The document is "normal" (prose-heavy, not heavily marked-up)
- You want text + tables without thinking about which library to use
- You're batch-processing many files
- You're feeding the output directly to another agent or LLM

**Deep mode — the workflow below**
Step-by-step diagnosis and extraction. Use it when:
- The document has tracked changes or comments you care about
- You need to extract images or embedded media
- Quick mode failed, returned garbage, or skipped a region
- The file is a legacy binary `.doc` (needs conversion first)

Default to quick mode. Fall back to deep mode when quick mode isn't enough.

### Quick mode usage

```bash
# Extract everything
python3 <skill-root>/scripts/quick_extract.py document.docx

# Drop tables, keep only prose
python3 <skill-root>/scripts/quick_extract.py document.docx --no-tables

# Force a specific extractor
python3 <skill-root>/scripts/quick_extract.py document.docx --tool pandoc

# Read from stdin
cat document.docx | python3 <skill-root>/scripts/quick_extract.py -
```

The script:
- Returns Markdown on stdout (title, metadata, prose, tables)
- Returns diagnostics on stderr (which tool ran, warnings)
- Exits 0 on success, 1 on failure
- Auto-detects available tools and falls back through a chain
- Never pollutes stdout — the output is safe to pipe into another tool

The extractor chain tries, in order: **pandoc → python-docx → raw-XML walk over zipfile**. The first one that produces non-empty output wins.

## 2. Golden rule for deep mode: diagnose before you extract

Never run `python-docx` blindly on an unknown document. A document with tracked changes can hide paragraphs you never notice. A document generated by an older tool may have broken styling. A `.doc` with a `.docx` extension still fails `zipfile.ZipFile`. A two-second diagnostic step saves debugging garbage output.

Run this inspection block first:

```bash
# Is it actually a .docx? The file(1) signature tells you in one line.
file target.docx

# For a .docx, list the ZIP contents. What parts are inside?
unzip -l target.docx

# Quick textual sample: does pandoc parse it?
pandoc --from docx --to plain target.docx 2>/dev/null | head -30

# Core metadata (title, author, dates, revision count)
unzip -p target.docx docProps/core.xml | head -20

# Any tracked changes present? The presence of <w:ins> or <w:del> is the tell.
unzip -p target.docx word/document.xml | grep -cE '<w:(ins|del)\b' || true
```

What each output tells you:

- **`file`** — if it reports `Microsoft Word 2007+`, it is a `.docx` (ZIP). If it reports `Composite Document File V2 Document`, it is a legacy binary `.doc` and you must convert first.
- **`unzip -l`** — the part inventory is your map. Key parts: `word/document.xml` (main body), `word/comments.xml` (comments — usually absent when none), `word/footnotes.xml`, `word/endnotes.xml`, `word/media/` (embedded images), `word/settings.xml`, `word/styles.xml`, `docProps/core.xml` (metadata).
- **`pandoc … --to plain`** — if it produces readable prose, you can trust the pandoc path in quick mode. If it errors or produces gibberish, fall back to `python-docx`.
- **`docProps/core.xml`** — author, created/modified, revision count. Useful when a user asks "who wrote this" and "how many rounds did it go through".
- **Tracked-change count** — if greater than zero, a plain text extraction will hide either the "before" or the "after"; decide which you want and extract explicitly.

## 3. Pick a strategy based on what the document contains

Classify in your head, then apply the matching workflow.

### Prose-heavy documents (letters, memos, policies, contracts, briefs)

Primary tool: `pandoc`. It preserves lists, headings, footnotes and tracked-change sides with a single flag.

```bash
# Clean extraction (accepts all tracked changes)
pandoc --from docx --to markdown document.docx --output document.md

# Show both sides of tracked changes inline
pandoc --from docx --to markdown --track-changes=all document.docx
```

If `pandoc` is not available, use `python-docx`:

```python
from docx import Document

doc = Document("document.docx")
paragraphs = [p.text for p in doc.paragraphs if p.text.strip()]
full_text = "\n\n".join(paragraphs)
```

### Documents with tracked changes

Symptom: the `<w:ins>` / `<w:del>` count from the diagnostic block is nonzero.

`pandoc --track-changes=<mode>` gives you three lenses:
- `accept` — keep insertions, drop deletions (what you get if you accept every revision)
- `reject` — drop insertions, keep deletions (the "before" state)
- `all` — keep both, annotated inline

Pick explicitly based on what the user asked for. Do not assume.

### Documents with comments

Symptom: `word/comments.xml` is present in the `unzip -l` listing.

`pandoc` preserves comments in markdown-extended form. For structured extraction (author, anchor, text), read the XML directly:

```python
import zipfile
from lxml import etree

NS = {"w": "http://schemas.openxmlformats.org/wordprocessingml/2006/main"}

with zipfile.ZipFile("document.docx") as z:
    with z.open("word/comments.xml") as f:
        tree = etree.parse(f)

for c in tree.getroot().findall("w:comment", NS):
    author = c.get("{http://schemas.openxmlformats.org/wordprocessingml/2006/main}author")
    text = "".join(t.text or "" for t in c.iter(f"{{{NS['w']}}}t"))
    print(f"{author}: {text}")
```

### Table-heavy documents (reports with data, invoices, statements)

`python-docx` gives you cell-level access:

```python
from docx import Document

doc = Document("report.docx")
for ix, table in enumerate(doc.tables, 1):
    rows = []
    for row in table.rows:
        rows.append([cell.text.strip() for cell in row.cells])
    # rows[0] is usually the header row
    print(f"Table {ix}: {len(rows)-1} data rows")
```

Caveat: merged cells appear as the same cell object in every merged position. If you are reconstructing a DataFrame, deduplicate by `id(cell)` per row before treating the value as distinct.

### Documents with embedded images

Images live inside the ZIP under `word/media/`. Extract them directly, no library required:

```python
import zipfile
from pathlib import Path

out_dir = Path("/tmp/docx_media")
out_dir.mkdir(exist_ok=True)

with zipfile.ZipFile("document.docx") as z:
    for name in z.namelist():
        if name.startswith("word/media/"):
            data = z.read(name)
            (out_dir / Path(name).name).write_bytes(data)
```

If you also want the image's position and caption, walk `document.xml` looking for `<w:drawing>` elements and their neighbouring `<w:p>` paragraphs.

### Legacy binary `.doc` files

Symptom: `file` reports `Composite Document File V2 Document`, or `zipfile.ZipFile` raises `BadZipFile`.

Convert with LibreOffice headless, then treat the result like any `.docx`:

```bash
soffice --headless --convert-to docx --outdir /tmp/docx_in target.doc
# /tmp/docx_in/target.docx is now a modern .docx
```

Without LibreOffice, the `.doc` is not readable with Python libraries in the baseline. Install `libreoffice` / `libreoffice-writer` — already part of the sandbox image.

### Mixed documents

Real-world files often mix everything: prose with tables interleaved, images, comments, maybe a tracked change on page three. Don't try to extract in one pass:

1. Extract prose with `pandoc` (decide the tracked-change mode first).
2. Extract tables with `python-docx` in a second pass.
3. Pull images from `word/media/` with `zipfile` if the user asked for them.
4. Surface comments separately — they are metadata, not body content.
5. Join findings at the end as a structured report.

## 4. Token-awareness when feeding content into an LLM

A .docx with 80 pages of prose produces roughly 25,000 tokens of markdown. If you only need three paragraphs, find them first and extract only those:

```python
from docx import Document

doc = Document("long_doc.docx")
target_heading = "Retention Policy"
capturing = False
collected = []
for p in doc.paragraphs:
    style = (p.style.name or "").lower()
    if "heading" in style:
        capturing = (p.text.strip() == target_heading)
    elif capturing:
        collected.append(p.text)
```

Rough token cost:
- Prose-only extraction: ~200 tokens per page
- Prose + tables (as markdown): ~350–500 tokens per page
- Embedded images: zero tokens unless the agent reads them multimodally

## 5. Metadata

`docProps/core.xml` and `docProps/app.xml` carry the interesting metadata:

```python
import zipfile
from lxml import etree

NS = {
    "cp": "http://schemas.openxmlformats.org/package/2006/metadata/core-properties",
    "dc": "http://purl.org/dc/elements/1.1/",
    "dcterms": "http://purl.org/dc/terms/",
}

with zipfile.ZipFile("document.docx") as z:
    with z.open("docProps/core.xml") as f:
        core = etree.parse(f).getroot()

def _get(xpath):
    node = core.find(xpath, NS)
    return node.text if node is not None else None

print({
    "title": _get("dc:title"),
    "author": _get("dc:creator"),
    "created": _get("dcterms:created"),
    "modified": _get("dcterms:modified"),
})
```

## 6. Cleaning extracted output

Raw `python-docx` extraction usually needs a small pass:
- Collapse runs of blank lines to at most two
- Strip zero-width characters (`\u200b`, `\ufeff`) that `pandoc` removes but `python-docx` preserves
- Normalise smart quotes to straight quotes if the downstream consumer expects ASCII (only when that is explicitly requested)
