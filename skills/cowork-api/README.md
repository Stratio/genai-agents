# cowork-api

Shared skill that wraps the Stratio Cowork management REST API (`genai-api`) so other skills and agents can register, deploy, and manage skills and agents directly from inside the Stratio sandbox without the manual download-and-upload roundtrip through the web UI.

Designed as a **router**: a minimal `SKILL.md` lists the available capabilities and points to one self-contained file under `tasks/` per capability. New endpoints are added by dropping a new file there — the index grows linearly without inflating the entry point.

## What it does

Today:

- **Upload a skill bundle** — `POST /v1/agents/skills/bundle/import` with a multipart ZIP. Supports `on_conflict` strategies (`new_version`, `overwrite`, `fail`) and an optional explicit semantic version.
- **Upload an agent bundle** — `POST /v1/agents/bundle/import` with a multipart `agents/v1` container ZIP. Reads `name` from the bundle's `metadata.yaml`. Supports the same `on_conflict` strategies.

Tomorrow (extension points): listing/exporting versions, publishing draft versions, managing projects, registering MCP servers, model groups — anything the `genai-api` REST surface exposes.

## When to use

Load this skill when the agent needs to **call the Cowork management plane** from inside the Stratio sandbox: registering a freshly packaged skill or agent, querying or mutating their lifecycle, etc.

This skill is **not** about runtime data access (those are the `stratio_data` / `stratio_gov` MCPs) and **not** about LiteLLM model calls (that goes through the internal proxy at `127.0.0.1:3002`, not through `genai-api`).

## Shared guides

- `external-api-calls.md` — declared in `guides`. Covers env vars (`GENAI_API_URL`, `USER_CERT_PATH`, `USER_KEY_PATH`, `CA_CERT_PATH`), the standard pre-check, and the curl/Python templates this skill builds on. Every sub-file under `tasks/` references it for the cross-cutting bits (mTLS, error handling).

## MCPs

None. This skill talks directly to `genai-api` over HTTPS with mTLS — no MCP server is involved.

## Python dependencies

None for the shipped templates (everything is bash + curl). If a caller prefers Python, the templates in `external-api-calls.md` use `requests` or `httpx`, both already present in `/opt/venv` inside the sandbox.

## System dependencies

- `curl` — present in the sandbox image.
- `~/.curlrc` is auto-generated by the sandbox at start with the client cert/key, so plain `curl` already negotiates mTLS; this skill still adds `--cert`, `--key`, and `--cacert` explicitly so the same script also works outside the sandbox if the env vars are set.

## Notes

- All endpoints are relative to `${GENAI_API_URL%/}`. The skill expects that variable to be set; otherwise the pre-check stops the call cleanly.
- Authentication is mTLS only. The user's certificate identity also drives RBAC/whitelist authorization on the server side — a 401/403 response means the user's role does not allow the operation, surface the body verbatim so the missing role is visible.
- This skill never silently retries. A failed call is reported to the caller with HTTP code and body; deciding whether to retry is the caller's responsibility.
