# Stratio MCP Usage Guide

## 1. Fundamental Rule

**NEVER write SQL manually.** The MCP system has a sophisticated query generation engine that understands the governed domain, its business rules, table relationships, and constraints. Always delegate query generation and execution to the MCP.

## 2. Available MCP Tools

| Step | MCP Tool | Purpose |
|------|----------|---------|
| 1a | `search_domains(search_text, domain_type?, refresh?)` | **Prefer over `list_domains`**. Search domains by free text (name or description). Results sorted by relevance. Use when part of the domain name or topic is known. `domain_type`: `'business'` (published semantic domains, with business names), `'technical'` (raw data domains, with database identifiers) or `'both'` (default — all). `refresh`: cache bypass |
| 1b | `list_domains(domain_type?, refresh?)` | List all available domains. Use only when you need to see all domains without filtering or when `search_domains` returns no results. Same `domain_type` and `refresh` parameters as `search_domains` |
| 2 | `list_domain_tables` | Get the domain's tables |
| 3 | `get_tables_details` | Understand business rules and context |
| 4 | `get_table_columns_details` | Get columns, types, and meanings |
| 5 | `search_domain_knowledge` | Understand terminology and definitions |
| 6 | `query_data` | **Retrieve data** (natural language question -> data) |
| 7 | `generate_sql` | View the SQL before executing (optional, for review) |
| 8 | `execute_sql` | Re-execute SQL generated by the MCP (never manual SQL) |
| 9 | `profile_data` | Quick statistical EDA |
| 10 | `propose_knowledge` | Propose discovered business terms |

## 3. Strict Rules

- **IMMUTABILITY of `domain_name`**: The `domain_name` parameter in ALL MCP calls must be **exactly** the value returned by `list_domains` or `search_domains`. NEVER translate, interpret, paraphrase, or infer it. If the domain is called `semantic_AnaliticaBanca`, use `"semantic_AnaliticaBanca"` — not `"Banca Particulares"`, not `"Analítica Banca"`, not `"banca"`. If in doubt about the exact name, call `search_domains` or `list_domains` again to confirm
- NEVER write SQL queries directly. Always use `query_data` or `generate_sql`
- For simple aggregations (totals, averages, counts): use `query_data` directly
- For advanced analysis: always try to resolve with `query_data` first (the MCP supports joins, aggregations, window functions, subqueries). Use Python/pandas only when the calculation cannot be expressed in SQL (statistical tests, segmentation, iterative transformations, procedural logic). In that case, retrieve the data with `output_format="dict"` and process it in pandas
- **MCP-first**: Always resolve in the MCP everything that can be expressed as a SQL query. The MCP generates SQL that understands the governed domain, its relationships, and business rules. Use Python/pandas ONLY for what SQL cannot solve: statistical tests, segmentation, iterative transformations, procedural logic, or data preparation for visualization. For multiple datasets:
  - **A single MCP query** when: the result requires data from several related tables (the MCP generates the JOINs), or aggregations with complex filters. Always try this first
  - **Multiple independent queries** when: you need orthogonal data slices (e.g., one temporal query + one query by segment + one ranking query). Launch in parallel
  - **Combine in pandas** only when: you need calculations that SQL cannot solve (statistics, segmentation) on data from several queries, or iterative transformations on transactional detail
  - **Inconsistencies**: If two queries yield different totals, verify granularity and filters. Reformulate with `additional_context` to align
- You can provide `additional_context` to the MCP to guide generation (e.g., business definitions, specific filters)
- **`output_format` is a string**: Valid values are `"dict"`, `"csv"`, or `"markdown"`. It is optional (default: `"dict"`). NEVER pass a boolean (`true`/`false`). If you don't need a specific format, omit the parameter
- If a query fails or returns unexpected results: reformulate the question in natural language, do not try to write SQL
- **Profiling (`profile_data`)**: Requires SQL as a parameter — ALWAYS generate it with `generate_sql`, never write it manually. NEVER add LIMIT to the SQL; use the tool's `limit` parameter instead
- **Parallel execution**: When the plan defines multiple independent data questions (none needs the result of another to be formulated), launch ALL `query_data` calls in a single response so they execute in parallel. This also applies to metadata calls (`get_table_columns_details`, `profile_data`, etc.). Only serialize when one query depends on the result of another (e.g., you need a value from query A to formulate query B)

## 4. Domain Discovery Workflow

Steps to explore a governed domain and understand its data before analysis.

### 4.1 Discover Domains

**Prefer searching over listing** — `search_domains` returns relevant results without loading the full list (which can be very large).

If the user provides a domain or gives hints about the topic:
- Execute `search_domains(name_or_hint)` to search for matches
- If it matches a result -> use it directly and go to step 4.2
- If there are no matches -> execute `list_domains()` as a fallback and ask the user

If there is no clear domain, ask the user which one they are interested in. If the user gives no hints, execute `list_domains()` to show all available domains (present as selectable options).

If a recently published or created domain does not appear, retry with `refresh=true` (cache bypass).

### 4.2 Explore Tables

1. `list_domain_tables(domain_name)` to list all tables in the domain
2. Present the tables with their descriptions in markdown table format

### 4.3 Table Details

For the tables of interest:
1. `get_tables_details(domain_name, table_names)` to obtain:
   - Full description
   - Business context
   - Associated business terms
   - Business rules
   - SQL behaviors
2. Present the information in a structured format

### 4.4 Columns

For each table of interest:
1. `get_table_columns_details(domain_name, table_name)` to obtain name, type, and business description
2. Present in a logically ordered markdown table

**Launch in parallel** steps 4.3 and 4.4 when they are about independent tables. Also launch step 4.5 in parallel if the terms to search are already known.

### 4.5 Business Terminology

`search_domain_knowledge(question, domain_name)` to search for:
- Business term definitions
- Calculation rules
- Data policies
- Domain glossary

## 5. Statistical Profiling

For `profile_data` usage rules (generate SQL with `generate_sql`, never manual SQL, use the `limit` parameter instead of LIMIT in SQL), see section 3.

Adaptive profiling thresholds based on estimated size:

| Estimated rows | Strategy | limit parameter |
|---------------|----------|-----------------|
| <100K | Full | Do not set (default) |
| 100K - 1M | Sampling | `limit=100000` |
| >1M | Sampling + alert | `limit=100000` + inform the user |

Document in reasoning if sampling was used.

## 6. MCP Clarification Responses

`query_data` and `generate_sql` may respond with a clarification request
instead of data (e.g., "What period are you referring to?", "Does 'Active' include users with a purchase
in 30 or 90 days?"). This is not an error — it is the engine requesting additional context.

Cascading protocol (follow in order):
1. **Search in the domain**: Call `search_domain_knowledge` with the ambiguous term.
   If the definition is found, re-call with `additional_context` including the definition
2. **Infer from the plan**: If the analysis plan already defines the term or period, add it
   directly to `additional_context` and re-call
3. **Ask the user**: Only if steps 1-2 do not resolve the ambiguity. Present
   the question with concrete options (never free text if there are clear options)
4. **Reformulate**: If the ambiguity persists, reformulate the data question with greater
   specificity (explicit dates, definitions embedded in the text)
5. **Report and continue**: If the MCP cannot respond after these steps, document
   the limitation and continue the analysis with the available data

Maximum 2 clarification iterations per query. If after both iterations there is no data,
inform the user and omit that metric from the analysis.

## 7. Post-Query Validation

Each `query_data` result must pass these 7 validations before being used in the analysis. When queries are launched in parallel, validate each result as it is received:
1. **Non-empty dataset** (>0 rows). If empty: reformulate the question or alert the user
2. **Expected columns present**. If missing: review the question formulation
3. **Coherent data types** (dates are dates, numerics are numerics)
4. **Temporal range** covers the requested period
5. **Null proportion** in key columns (<50%). If exceeded: document the limitation
6. **Values in reasonable ranges** (no ages of 500 years, unexpected negative amounts)
7. **Business sanity check**: Verify that the results make sense:
   - Reasonable magnitudes (500% MoM growth is probably a data error)
   - Consistency with domain knowledge (`search_domain_knowledge`)
   - If a finding seems "too good/bad", investigate before reporting

If any validation fails: reformulate the question to the MCP, inform the user of the limitation, and adjust the plan if necessary.

## 8. Timeouts and Retries

If the MCP takes too long or returns an error:
1. **Simplify the question**: Reduce dimensions or time period
2. **Split the query**: Break a complex question into several simpler ones
3. **Reformulate**: Express the same question differently
4. Do not retry the same question more than 2 times — if it persists, inform the user

## 9. Best Practices for Formulating Questions

- **Be specific with periods**: "monthly sales for the last year" instead of "sales"
- **Include dimensions**: "by region and product category"
- **Specify metrics**: "total revenue and number of transactions"
- **Use additional_context**: For non-obvious definitions (e.g., "Active customers = at least 1 purchase in 90 days")
- **One question = one dataset**: Do not mix unrelated questions
- **Think about granularity**: Do I need aggregated data or transactional detail?

**Query strategy** — PLANNING order (think from general to specific):
1. **General context**: Totals, basic counts -> understand the dataset magnitude
2. **Dimensional queries**: By time, segment, region -> find patterns and trends
3. **Detail queries**: Top/bottom N, outliers -> dig deeper into findings
4. **Validation queries**: Data cross-checks, consistency checks -> ensure reliability

This order is for **planning** the questions. In **execution**, launch all independent queries in parallel — typically categories 1, 2, and 3 can be executed simultaneously. Only category 4 (cross-validation) may require previous results.
